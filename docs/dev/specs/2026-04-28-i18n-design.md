# Internationalization (i18n) Design

**Date:** 2026-04-28  
**Status:** Approved  
**Scope:** Full-stack i18n framework — Chinese + English, extensible to additional languages

---

## Overview

cubeplex currently has mixed hardcoded strings (Chinese in sidebar/admin pages, English in auth forms). This design establishes a proper i18n framework: next-intl on the frontend, Babel/gettext on the backend, with language preference stored in the user's account.

---

## Architecture

```
Frontend (next-intl)
  middleware.ts
    ├── Authenticated: read NEXT_LOCALE cookie → locale
    └── Unauthenticated: parse Accept-Language header → locale, fallback: en

  messages/en.json
  messages/zh.json

  Server components → getTranslations()
  Client components → useTranslations()

        │ API requests + Accept-Language: <locale> header
        ▼

Backend (Babel + gettext)
  User.language field (en | zh, default: en)
  PATCH /api/v1/auth/me  ← update language preference

  locale dependency → parse Accept-Language from request header
  cubeplex/i18n/messages/en/LC_MESSAGES/messages.po
  cubeplex/i18n/messages/zh/LC_MESSAGES/messages.po

  FastAPI exception handlers → localized error messages
```

**Key conventions:**
- Supported locales: `en`, `zh` (extensible — add locale files to expand)
- `NEXT_LOCALE` cookie is the single source of locale truth on the frontend
- Backend is stateless with respect to locale — reads `Accept-Language` header on every request
- `ApiClient` in `@cubeplex/core` automatically adds `Accept-Language` header based on current locale

## Backend Changes

### 1. User model — `language` field

Add `language: str = Field(default="en", max_length=10)` to the `User` SQLModel.

- Validated via Pydantic enum (`Literal["en", "zh"]`) on write
- Returned in `GET /api/v1/auth/me` response (no route change needed)
- One Alembic autogenerate migration

### 2. Language preference endpoint

```
PATCH /api/v1/auth/me
Body: { "language": "zh" }
Response: updated user object
```

Validates that `language` is in the supported set. Authenticated only.

### 3. i18n translation layer

Directory: `cubeplex/i18n/`

```
cubeplex/i18n/
├── __init__.py          # get_translator(locale) → lazy gettext Callable
└── messages/
    ├── en/LC_MESSAGES/messages.po   (+ compiled .mo)
    └── zh/LC_MESSAGES/messages.po   (+ compiled .mo)
```

`get_locale` FastAPI dependency: reads `Accept-Language` header, extracts first matching supported locale, falls back to `en`.

### 4. Localized error messages

All user-visible `HTTPException` details and business exceptions go through `gettext`. Translation keys use snake_case short strings (e.g., `not_member`, `invalid_credentials`). `.po` files map keys to natural-language messages in each locale.

Example:
```python
# Before
raise HTTPException(status_code=403, detail="Not a member of this workspace")

# After
raise HTTPException(status_code=403, detail=_("not_member"))
```

## Frontend Changes

### 1. next-intl setup

New files in `packages/web/`:
```
messages/
  en.json
  zh.json
i18n/
  request.ts    # next-intl server config — reads NEXT_LOCALE cookie
middleware.ts   # extended with locale detection logic
```

`next.config.ts` updated with next-intl plugin.

### 2. Translation file namespaces

```json
{
  "auth":      { "signIn": "...", "signUp": "...", "email": "...", ... },
  "sidebar":   { "newChat": "...", "recentChats": "...", "newWorkspace": "...", ... },
  "workspace": { "title": "...", ... },
  "admin":     { "skills": "...", ... },
  "errors":    { "notMember": "...", "invalidCredentials": "...", ... },
  "common":    { "save": "...", "cancel": "...", "loading": "...", ... }
}
```

All existing hardcoded strings (Chinese and English) are extracted into these files.

### 3. ApiClient locale injection

`ApiClient` in `@cubeplex/core` accepts a `locale` parameter at construction. Every outgoing request adds `Accept-Language: <locale>` header. Frontend passes `useLocale()` value when creating API clients.

### 4. Login flow — locale sync

```
Login success
  → GET /auth/me → { ..., language: "zh" }
  → Write NEXT_LOCALE=zh via document.cookie (consistent with existing CSRF cookie pattern)
  → router.refresh()  ← next-intl middleware picks up new cookie
```

Logout clears the `NEXT_LOCALE` cookie; middleware falls back to `Accept-Language`.

### 5. Language switcher

Located in `AvatarPopover.tsx` — two toggle buttons (EN / 中文):

```
Click "中文"
  → PATCH /api/v1/auth/me { language: "zh" }
  → Write NEXT_LOCALE=zh
  → router.refresh()
```

Unauthenticated users: switcher writes cookie only (no API call).

## Data Flows

### Authenticated — login
```
POST /auth/login
→ GET /auth/me → { language: "zh" }
→ frontend: NEXT_LOCALE=zh cookie
→ router.refresh()
→ middleware reads cookie → locale = zh globally
```

### Authenticated — language change
```
User selects EN
→ PATCH /auth/me { language: "en" }
→ NEXT_LOCALE=en cookie
→ router.refresh()
→ all subsequent API requests: Accept-Language: en
```

### Unauthenticated
```
GET /login (no NEXT_LOCALE cookie)
→ middleware reads Accept-Language: zh-CN
→ matches zh → locale = zh (cookie not persisted to DB)
→ on login: user.language overrides
```

### Edge cases

| Scenario | Handling |
|----------|----------|
| Cookie cleared | Middleware re-reads Accept-Language, seamless |
| `user.language` = unsupported future value | Frontend falls back to `en`, no crash |
| Missing Accept-Language on API request | Backend defaults to `en` |
| Logout | Clear NEXT_LOCALE cookie → revert to browser language |

---

## Out of Scope (this iteration)

- URL-prefix based routing (`/zh/...`, `/en/...`)
- RTL language support
- Plural/gender rules (ICU format) — simple key-value sufficient for zh/en
- Translation management tooling (Lokalise, Crowdin, etc.)
- Automatic string extraction CI checks
