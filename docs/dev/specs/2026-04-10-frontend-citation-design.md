# Frontend Citation System Design

## Overview

Display inline citation references from the agent's response text, with hover cards showing source metadata and click-to-open in the side panel with chunk highlighting. Citations use `【N-M】` markers generated by the backend's CitationMiddleware.

## Goals

- Parse `【N-M】` markers in assistant text and render as inline `N.M` pills
- Hover shows a card with favicon, title, chunk snippet, domain, date, URL
- Click opens the existing ToolDetailPanel with the cited chunk highlighted and scrolled into view
- Citation metadata persists via `ToolMessage.additional_kwargs["citations"]` in checkpoint — no new API endpoints needed
- Works for both live streaming and history loading

## Non-Goals

- Footnote/references section at the bottom of messages
- Citation grouping or deduplication in the UI
- Citation editing or user-added citations

---

## Data Flow

### Live Streaming

```
Backend SSE
  ├─ citation event ──→ citationStore[conversationId][citationId] = CitationData
  ├─ text_delta (with 【N-M】) ──→ messageStore (existing flow)
  └─ tool_result ──→ toolResultMap (existing flow, uses original_content)
```

### History Loading

```
GET /conversations/{id}/messages
  └─ tool message has citations field ──→ citationStore[conversationId] bulk load
  └─ assistant message content has 【N-M】 ──→ parsed at render time
```

### Backend Persistence (small backend change required)

`CitationMiddleware.awrap_tool_call` already stores `original_content` in `ToolMessage.additional_kwargs`. It should additionally store:

```python
result.additional_kwargs["citations"] = all_citation_data  # list[dict]
```

`convert.py` extracts this in the ToolMessage → API response mapping:

```python
citations = (msg.additional_kwargs or {}).get("citations")
```

This keeps citation metadata in the checkpoint, surviving cancel/disconnect.

## Citation Store (`packages/core/src/stores/citationStore.ts`)

```ts
interface CitationStore {
  // conversationId → citationId → CitationData
  citations: Record<string, Record<number, CitationData>>

  addCitation: (conversationId: string, data: CitationData) => void
  loadCitations: (conversationId: string, citations: CitationData[]) => void
  getCitation: (conversationId: string, citationId: number) => CitationData | undefined
  clearConversation: (conversationId: string) => void
}
```

- `addCitation`: called when a `citation` SSE event arrives during streaming
- `loadCitations`: called when loading history — iterates tool messages, extracts `citations` field, populates the store
- `getCitation`: used by `CitationMarker` and `CitationHoverCard` at render time

## Message Store Changes (`packages/core/src/stores/messageStore.ts`)

In the event processing pipeline, add a handler for the `citation` event type:

```ts
case 'citation':
  citationStore.getState().addCitation(conversationId, event.data as CitationData)
  break
```

In `loadMessages` (history), after loading messages, iterate tool messages and call `loadCitations` for any that have a `citations` field.

## Text Rendering — Marker Parsing

In `AssistantMessage.tsx`'s `ContentBlockRenderer`, for `text` blocks:

1. Split `block.content` by the regex `/【(\d+)-(\d+)】/g`
2. Produce an array of alternating text segments and `CitationMarker` components
3. Wrap with ReactMarkdown using a custom approach:
   - First pass: replace `【N-M】` with a placeholder token (e.g. `%%CITE:N:M%%`)
   - Render markdown normally
   - Use a custom `remarkPlugin` or post-process the rendered output to replace placeholders with `<CitationMarker>` components

**Simpler approach chosen**: Use ReactMarkdown's `components` prop to handle this via a custom text renderer that splits on citation patterns. ReactMarkdown already provides a `children` string for text nodes — intercept at that level.

```tsx
// In the text block renderer
<ReactMarkdown
  remarkPlugins={[remarkGfm]}
  components={{
    p: ({ children }) => <p>{renderWithCitations(children, conversationId)}</p>,
    li: ({ children }) => <li>{renderWithCitations(children, conversationId)}</li>,
    // ... other text-containing elements
  }}
>
  {block.content}
</ReactMarkdown>
```

`renderWithCitations` recursively walks React children, finds string nodes, splits on `【N-M】`, and inserts `CitationMarker` components.

- Small inline pill: rounded-full, semi-transparent primary background
- Text: `N.M` format (e.g., `3.0`, `3.1`)

## CitationHoverCard Component (`packages/web/components/chat/CitationHoverCard.tsx`)

A shadcn `HoverCard` popover displayed on marker hover.

### Layout (top to bottom)

1. **Header row**: Favicon (16px) + Domain text + Source type badge
   - Favicon: `https://www.google.com/s2/favicons?domain={domain}&sz=16`
   - On load error: replace with a default `Globe` icon from lucide
   - Domain: `text-xs text-muted-foreground`
   - Source type: tiny badge (e.g., "web", "file")

2. **Title**: `text-sm font-medium`, clickable (opens panel), line-clamp-2

3. **Chunk snippet**: `text-xs text-muted-foreground`, the cited chunk text, line-clamp-3 (~150 chars)

4. **Footer row**: Published date (if available) + URL (if web type, clickable → new tab)
   - Date: `text-xs text-muted-foreground/60`
   - URL: `text-xs text-primary truncate`, `target="_blank"`

### Sizing

- Max width: 320px
- Padding: 12px
- Card: `bg-popover border border-border rounded-lg shadow-lg`

## Side Panel Highlighting

### Panel Store Extension

Add optional `highlightText` to the tool panel view:

```ts
export type PanelView =
  | { type: 'closed' }
  | {
      type: 'tool'
      toolName: string
      toolArgs: Record<string, unknown>
      toolResult: string | null
      contentType: PanelContentType
      toolRef: ToolCallRef | null
      highlightText?: string | null // NEW: text to highlight and scroll to
    }
  | { type: 'artifact'; conversationId: string; artifactId: string }
```

`openTool` gains an optional `highlightText` parameter.

### Panel View Components

`SearchResultView`, `WebFetchView`, and `GenericToolView` accept an optional `highlightText` prop:

1. On mount/update, if `highlightText` is set:
   - Find the DOM element containing that text (using a `data-*` attribute or text content search)
   - Scroll it into view (`scrollIntoView({ behavior: 'smooth', block: 'center' })`)
   - Apply a temporary highlight style: `bg-yellow-200/40 dark:bg-yellow-500/20` with a fade-out transition (2s)

2. For `SearchResultView` specifically: each result item should be rendered with a `data-result-index` attribute. The citation's `tool_call_id` maps to the tool result, and the chunk content can be fuzzy-matched against result descriptions/content to find the right item to highlight.

## Edge Cases

- **No citation data in store**: If `getCitation` returns undefined (data not yet loaded or missing), render `【N-M】` as plain text — no pill, no hover.
- **Missing metadata fields**: All fields except `source_type` are optional. Hover card gracefully hides missing sections.
- **Favicon load failure**: `onError` handler hides `<img>` and shows `Globe` icon fallback (same pattern as existing `SearchResultView`).
- **Streaming text splits**: Backend already buffers incomplete `【...】` markers in SSE, so the frontend always receives complete markers.
- **Cancel mid-stream**: Citations already in checkpoint via `additional_kwargs["citations"]` survive. Partial text with markers still renders — any marker whose citation data is missing falls back to plain text.
