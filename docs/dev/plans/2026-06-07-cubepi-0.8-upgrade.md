# cubepi 0.8.0 Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bump cubepi from `v0.7.0` to `v0.8.0`, run the checkpointer v3→v4 schema migration, adopt `run_id` synchronization between cubeplex and cubepi, and migrate the two pure-function builtin tools to the new `@tool` decorator. Fork APIs are intentionally **not** wired up in this PR.

**Architecture:** Four edges, mostly mechanical.

1. **Pin + lock.** Bump `cubepi` git rev to `v0.8.0`; regenerate `uv.lock`. No transitive surprises expected — 0.8 core deps unchanged from 0.7 (anthropic, openai, pydantic).
2. **Alembic v3→v4.** New revision calls `upgrade_v3_to_v4_op()` (adds `run_id` column + index on `cubepi_messages`, creates partitioned `cubepi_runs` parent + 64 hash partitions) and `write_schema_version_op()` (bumps to 4). Update `alembic/env.py` to exclude `cubepi_runs` and `cubepi_runs_p*` from autogenerate, the same way `cubepi_messages` / `cubepi_messages_p*` are excluded.
3. **`run_id` sync.** cubeplex already owns a `run_id` (uuid7, propagated via redis run-meta hash, SSE streams, billing events, scheduled-task rows). 0.8's `Agent.prompt(run_id=...)` lets cubepi stamp the same id onto `cubepi_messages.run_id` instead of generating its own. Pass cubeplex's `run_id` at the **main `agent.prompt(...)` call site only** — `run_manager.py:1477` (`_execute_run`). `agent.respond(...)` does **not** accept `run_id` — it auto-recovers via `Checkpointer.load_pending` (already implemented upstream on `PostgresCheckpointer`), so HITL resume keeps the original `run_id` automatically. Reflection runner is **not** touched (see Risk Register #2).
4. **`@tool` decorator** (small cleanup). Convert the two genuinely pure-function tools (`datetime_tool`, `calculator_tool`) to the `cubepi.tool` decorator using **flat parameters with `Field(...)` metadata** (not a nested input model). All other builtin tools use the closure-builder pattern with bound dependencies (`org_id`, `workspace_id`, `session`, `objectstore`, capability bag) and the longhand `AgentTool(...)` constructor stays put.

**Not in scope:**
- `Agent.fork()` / `Agent.fork_once()` — no product surface for it yet.
- `Agent(messages=...)` pre-seed — reflection_runner and subagent middleware don't need it; both already seed history via the cubepi-owned channel.
- `state.active_run_id` exposure — cubeplex carries its own `run_id` via closure, no need to read it back from cubepi state.

**Behavior changes that affect cubeplex but require no code edit:**
- `AgentToolResult(is_error=True)` is now honored on the execute path. Previously (0.7) it was silently treated as a successful result. cubeplex has 13 such call sites (`view_images`, `load_skill`, `generate_image`, `artifacts`, `todo`, `actions/builder`, `run_manager`); audit confirms all are genuine errors that the model should see as errors. Verification: run the HITL + builtin-tool E2E suite and confirm no regressions in model retry/recovery behavior.
- `ask_user_tool` now populates `HitlBinding` on the returned `AgentTool`. cubeplex does not branch on `tool.hitl`, so no code change. Smoke-test the HITL pause/resume path.

**Tech Stack:** Python 3.13 / `uv` / cubepi 0.8.0 / `asyncpg`-backed cubepi PostgresCheckpointer / FastAPI / pytest.

**Create:**
- `backend/alembic/versions/<auto>_cubepi_v3_to_v4_runs.py` — generated via `uv run alembic revision --autogenerate -m "cubepi v3 to v4 runs"`. Because `cubepi_messages` is in `_CHECKPOINT_TABLES` and the new `cubepi_runs` parent + partitions are excluded by the env.py change in Task A3, autogen emits an empty `upgrade()` / `downgrade()` body. The exclusion is deliberate: SQLAlchemy `MetaData` cannot model partition DDL, so the upstream-supplied helper is the only correct source. Hand-write the body:
  ```python
  from cubepi.checkpointer.postgres.alembic_helpers import (
      upgrade_v3_to_v4_op,
      write_schema_version_op,
  )

  def upgrade() -> None:
      # Hand-written: cubepi_messages + cubepi_runs are excluded from autogenerate
      # (they're partitioned tables — SQLAlchemy MetaData can't emit the DDL).
      # The helper adds run_id+index to cubepi_messages, creates the partitioned
      # cubepi_runs parent + 64 hash partitions, and is idempotent (IF NOT EXISTS).
      op.execute(upgrade_v3_to_v4_op())
      op.execute(write_schema_version_op())  # bumps cubepi_schema_version to 4

  def downgrade() -> None:
      # Symmetric drop — DROP TABLE CASCADE removes child partitions and the
      # ix_cubepi_runs_thread_seq index along with the cubepi_runs parent.
      # Dropping the run_id column auto-drops ix_cubepi_messages_thread_run
      # (the index references run_id), but we DROP IF EXISTS first for clarity.
      op.execute("DROP TABLE IF EXISTS cubepi_runs CASCADE")
      op.execute("DROP INDEX IF EXISTS ix_cubepi_messages_thread_run")
      op.execute("ALTER TABLE cubepi_messages DROP COLUMN IF EXISTS run_id")
      op.execute(
          "DELETE FROM cubepi_schema_version WHERE version <> 3; "
          "INSERT INTO cubepi_schema_version (version) VALUES (3) "
          "ON CONFLICT DO NOTHING;"
      )
  ```
  **Artifacts the helper creates** (verified against `cubepi/checkpointer/postgres/alembic_helpers.py:upgrade_v3_to_v4_op()`):
  - `cubepi_messages.run_id` column (`ADD COLUMN IF NOT EXISTS`)
  - `ix_cubepi_messages_thread_run` index on `cubepi_messages (thread_id, run_id)` (`CREATE INDEX IF NOT EXISTS`)
  - `cubepi_runs` partitioned parent (`CREATE TABLE IF NOT EXISTS`), with FK `(thread_id)` → `cubepi_threads ON DELETE CASCADE`
  - `ix_cubepi_runs_thread_seq` index on `cubepi_runs (thread_id, completion_seq)` (`CREATE INDEX IF NOT EXISTS`)
  - 64 child partitions `cubepi_runs_p00..p63` via `create_runs_partitions_op()` — **partition `CREATE TABLE` statements do NOT use `IF NOT EXISTS`**, so re-running the migration after a partially-failed first run requires manual cleanup of any orphaned partitions before retry.

  **Why this is `--autogenerate`-compatible (per AGENTS.md hard rules):** the workflow uses `alembic revision --autogenerate` — the body is empty after the env.py exclusion in A3 is in place. Hand-adding helper-driven DDL to an autogenerated revision is the same pattern the previous migration `060310ecfd8a` used for `write_schema_version_op()`. AGENTS.md "do not hand-edit migration files" rule prohibits forging schema diffs autogen would catch; it does not prohibit appending excluded-table DDL or data rows that autogen structurally cannot emit.

**Verify, no edit expected:**
- `backend/cubeplex/agents/checkpointer.py` — thin wrapper around `PostgresCheckpointer`. Upstream 0.8 already implements the v4 Protocol methods (`snapshot` / `fork` / `claim_run` / `mark_run_complete` / `load_pending`). The wrapper is opaque to those.
- `backend/cubeplex/streams/run_manager.py:1859` `await agent.respond(...)` — no edit; respond auto-recovers `run_id` via `load_pending`. The recovered id equals the original because cubeplex-side `run_id` flowed through `prompt()` and is now persisted on the pending HITL row.
- All other `Agent(...)`, `Model(...)`, middleware, provider call sites — already on the 0.7 bound-model API.

## Open Questions / Risk Register

1. **`is_error=True` semantic flip.** 0.7 silently dropped the flag; 0.8 reports the tool result as an error to the model. cubeplex's 13 call sites all return error text in `TextContent`, so the model already saw the error string. The only behavioral delta is that providers can now key off `is_error=True` for retry / formatting (Anthropic `tool_result.is_error=true`). Risk: model may retry harder or stop earlier. Mitigation: HITL + builtin-tool E2E suite must pass with no regressions.
2. **Reflection runner explicitly skipped.** Earlier draft of this plan also stamped `inp.run_id` on `reflection_runner.py:122`. Verified that the reflection agent is built by `_make_reflection_agent` (in `run_manager.py`) **without a `checkpointer` or `thread_id`** — it's a transient in-memory agent that only emits `tool_execution_end` events for the listener to scrape. Nothing it does is appended to `cubepi_messages` (or any DB table). Passing `run_id=inp.run_id` would have no DB effect, and if the agent were ever upgraded to use a checkpointer with the same `thread_id`, reusing the parent's already-completed `run_id` would trigger `claim_run()` failure on `(thread_id, run_id)` collision. So reflection is **not** modified in this PR. If we later want reflection writes persisted, the change needs its own design (separate `thread_id` or a derived run id like `f"{inp.run_id}:reflection"`).
3. **Migration ordering.** New revision's `down_revision` must point at the latest head (the most recent migration on `main`). Verify with `alembic heads` inside the worktree before writing the file.
4. **Partial uv.lock churn.** `uv lock` may pull in newer transitive pins beyond cubepi. Inspect the lock diff before committing — only `cubepi` and its direct transitive change set should move.

## Rollback

If the migration runs but agent runtime regresses:
1. `uv run alembic downgrade -1` reverts schema (the cubepi 0.7 pin can still run against v3 because v3 is a strict subset of v4 — the column/table are simply absent).
2. Revert the pin bump and `uv.lock`.
3. Revert the cubeplex source edits (run_id sync + `@tool` conversion).
The four edges land in one commit, so a single `git revert` covers all of them.
