# Invite Redesign + Accept-Page Fixes — Implementation Plan

**Date:** 2026-07-01
**Branch:** `feat/2026-06-29-registration-flow` (follow-up to registration flow)

## Background

Manual testing of the registration flow surfaced three issues with the invite
system. This plan addresses all three; they are coupled (problem 3 removes the
ws-invite surface that problems 1 & 2 currently live on), so problem 3 is done
first.

## Problems

1. **Invite-link origin mismatch.** The emailed link uses the backend
   `frontend_base_url` (localhost:3001), but the on-screen link in
   `CreateInviteDialog` uses `window.location.origin` (192.168.x.x:3001). The
   two disagree. Decision: unify on the backend `frontend_base_url` — links are
   generated by the backend and returned to the frontend for display.

2. **"Invite already used" on first open for a logged-in user.** The
   `(auth)/invite/accept` page fires `acceptInvite` in a `useEffect` with no
   re-entry guard. React StrictMode (dev) double-mounts the effect → two
   concurrent `POST /invites/accept` calls → one 200, one 400 (token already
   consumed), and the user sees the 400. Backend log confirms two requests
   30ms apart.

3. **Workspace invites cross org boundaries.** `accept_invite` grants
   `OrgRole.MEMBER` on the ws's org, so a workspace invite can pull an
   outsider into the org. Decision: **remove workspace invites entirely**.
   Workspace members are added directly from the org's existing members
   (`POST /ws/{ws}/members` + `GET /ws/{ws}/members/available` already exist).
   Inviting (link-based, for users not yet in the org) moves to the **org
   member admin page** via the org-invite endpoints added in the registration
   flow (`POST /admin/orgs/invites` + `POST /orgs/invites/accept`).

### Task D: Fix accept-page re-entry (problem 2) for the org accept page

**Files:**
- Modify: `frontend/packages/web/app/(auth)/orgs/invites/accept/page.tsx` — add a
  `useRef` re-entry guard so the effect calls `acceptOrgInvite` exactly once even
  under StrictMode double-mount. Pattern:

```ts
const calledRef = useRef(false)
useEffect(() => {
  if (calledRef.current) return
  calledRef.current = true
  // ... acceptOrgInvite ...
}, [token])
```

- [ ] Add the guard; verify no double `POST /orgs/invites/accept` in the backend
      log when opening the link as a logged-in user.

## Testing

- Org invite: org admin → create invite (member) → copy link → open in a
  different browser/not-logged-in → register/login → land in onboarding
  (workspace-only) → create ws → member of org+ws. Single `POST .../accept`

  backend `frontend_base_url`).

  `test_invite_onboarding.py`).
