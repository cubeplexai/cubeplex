# 在对话中使用工具

工作区中启用 MCP 连接器后，智能体可以自动调用其工具。无需为每个对话单独启用任何功能——所有已启用连接器默认均可用。

## 智能体如何决定使用工具

发送消息时，智能体会看到来自工作区已启用连接器的可用工具列表。如果你的请求能从外部数据或操作中受益，智能体会自行调用合适的工具。例如：

- “GitHub 上分配给我的未关闭 issue 有哪些？”会触发 GitHub 连接器。
- “搜索近期有关 transformer 架构的文章”会触发 Web 搜索连接器。
- “将此对话的摘要发布到 Slack”会触发 Slack 连接器。

你无需明确指定工具。智能体会将你的意图匹配到正确的连接器。不过，如果你希望强制使用特定工具，也可以具体说明：

> 使用 Exa 查找有关气候建模的学术论文。

## 阅读工具结果

智能体调用工具时，对话会显示：

1. **工具调用指示器** — 显示所调用工具及其输入摘要的标签。
2. **工具结果** — 外部服务返回的数据，会被整合到智能体的回复中。
3. **引用** — 显示提供信息的连接器和来源的徽章或注释。引用可帮助你验证数据的来源。

智能体会将工具结果融入自然语言回复。你会在文本旁看到工具调用详情，从而可将每项说法追溯到其来源。

## 多次工具调用

智能体可以在一个轮次中调用多个工具。例如，如果你请求：

> 将最新 GitHub issue 与我们的 Linear 待办事项进行比较，并总结重叠部分。

智能体可能会同时调用 GitHub 和 Linear 连接器，然后将结果整合为一条回复。

## 渐进式披露

当工作区启用了许多连接器时，CubePlex 会使用 **渐进式披露** 来保持智能体高效。当合并后的工具定义会占用模型上下文窗口过多空间时，系统会停止预先加载所有工具，改为：

1. 将每个连接器的工具分组为具名的延迟组，而不是将它们全部加载到智能体上下文中。
2. 向智能体展示每个组功能的轻量级摘要。
3. 仅当智能体判断某个连接器与请求相关时，按需加载该组的完整工具。

启用足够多连接器（至少需要两个才能激活）且工具相对于上下文窗口变得较大时，此机制会自动启动。无需进行任何配置——即使启用了数十个连接器，这项自动优化也能保持回复速度。

## 你可以请求智能体执行的操作

以下是各连接器类别的常见用法：

### 开发工具

- “显示 main 分支上的最近 5 个提交”（GitHub）
- “为登录 bug 创建一个新的 Linear issue”（Linear）
- “今天生产环境发生了哪些 Sentry 错误？”（Sentry）
- “查找有关我们部署流程的 Confluence 页面”（Atlassian）

### 生产力工具

- “列出我本周到期的 Notion 任务”（Notion）
- “检查我的 Asana 项目中是否有逾期项”（Asana）
- “向 #engineering 发送一条包含今日站会摘要的 Slack 消息”（Slack）

### Web 搜索

- “在 Web 上搜索最新的 Next.js 15 功能”（Tavily / Exa / WebTools）
- “查找近期有关多模态 LLM 的研究论文”（Exa）
- “阅读此 URL 的内容：……”（Jina AI / WebTools）

### 基础设施

- “检查我们域名的 Cloudflare 分析数据”（Cloudflare）
- “查询 example.com 的 DNS 记录”（Cloudflare）

## 故障排除

**智能体没有使用我预期的工具：**

- 确认连接器已启用：打开工作区 **MCP** 页面。
- 确保连接器未在工作区中被禁用（“已禁用”徽章表示组织管理员已在组织范围内暂停它）。
- 尝试在请求中写得更明确（例如，“使用 GitHub 连接器来……”）。

**智能体从工具获得错误：**

- **认证错误** — 连接器凭据可能已过期。请管理员重新连接（OAuth）或更新 API 密钥。
- **权限错误** — 凭据可能没有正确的范围。管理员可能需要使用更广泛的权限重新授权。
- **服务不可用** — 外部服务可能暂时宕机。请稍后重试。

**我没有看到任何工具调用：**

- 工作区可能没有启用任何连接器。打开工作区 **MCP** 页面进行检查，或请工作区管理员从目录中启用连接器。
- 你选择的模型可能不支持工具使用。请切换到支持工具使用的模型（大多数现代模型都支持）。
